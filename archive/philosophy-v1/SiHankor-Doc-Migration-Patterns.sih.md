# 文档重构模式：Legacy -> Docs

> 从 `legacy/plan/vs-plan.md` 的构建和执行中提炼的可复用经验。

## 核心原则

- **决策 != 道**：道是被发现的因果必然性，决策是被发明的工程选择。旧决策索引记录的是道的发现轨迹，不是道本身。本体论区分决定处置方式：废弃旧索引为"史"，不强行格式化为 ADR（"法"）。
- **审计前置删除后行**：删除 legacy 前必须逐文件审计迁移完整性。Git 历史提供可恢复性，但审计提供确定性。审计未通过 = 退回补充。
- **不创建占位文档**：知止之法：：尚未实现的功能（如 Mind）不应创建空白占位文档，空白占位是对目录结构的噪音污染。保留在 legacy，待工程化后迁移。
- **映射 vs 设计规范边界不可跨越**：Engineering-Mapping 的职责是"哲学概念->工程实践的投射"，不包含未经验证的具体配置值。展开配置值是从映射滑向设计规范。
- **1/3 文档不写入新内容**：未收敛的文档（stage 1/3）上做增量写入，每次对齐都需要重新检查一致性：：重复劳动。注记应写入已收敛的文档，或等目标文档收敛后交叉引用。

## 批量决策模式

12 项待决策用结构化表格一次性决议：每个决策点列出选项、决议、一句话理由。决议后写入 plan 的 $六（已决议事项表），后续执行直接引用决议编号（D1-D12），无需重新讨论。

## 遗留检测方法

标记为"废弃"的 legacy 文件在删除前，应检查每个废弃文件的独有主题词是否在 docs 中至少出现一次。通过 = 确认可删。不通过 = 补充迁移。不需要逐句比对（工作量和收益不成比例）。
