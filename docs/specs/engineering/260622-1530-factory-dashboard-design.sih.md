# 司衡工厂产线看板设计

> 司衡治理链的 UX 表达——装配线式产线图。内核不变，这是一层可选的视觉皮肤。

## 一、设计原则

### 1.1 不是隐喻——是结构同构

司衡的治理链结构上和一条装配线完全同构。不需要翻译，只需要画出：

| Factorio 概念 | 司衡概念                  | 映射方式                     |
| ------------- | ------------------------- | ---------------------------- |
| 传送带        | upstream 依赖链           | 流向即依赖方向，自动布局     |
| 分拣器        | stage 推进（1/3→2/3→3/3） | stage 关卡，通过后流向下一段 |
| 组装机        | 外部 Agent 生成           | 状态：等待/运行/完成         |
| 质检器        | validator                 | F=红灯停产，G=黄灯，J=日志   |
| 回收站        | 修正闭环                  | 违规文档回炉，修正后重入产线 |
| 电力          | 道/法元规则               | 全局约束，电力不足全厂限产   |
| 研究站        | 能力域解锁                | P0→P4 分期                   |

### 1.2 档案层始终存在

视觉层之下是真实的治理语义。任何视觉元素映射回标准术语：

- 所有按钮的 aria-label 包含治理术语
- 红色闪烁标注 F 级违规及编号
- 黄灯悬停显示 G 级违规详情

### 1.3 双重入口

同一个产线图，两种消费方式：

| 入口                    | 用户            | 体验                           |
| ----------------------- | --------------- | ------------------------------ |
| 浏览器 `localhost:9741` | 任何人          | 全功能——动画、可操作、实时推送 |
| MCP `image` 类型返回值  | 任何 Agent 用户 | 产线快照 PNG，30 秒刷新        |

## 二、产线架构

### 2.1 四条主产线

```text
┌─ spec 产线 ─────────────────────────────────────────┐
│ [追问机] → [spec组装机] → [质检器] → [仓库]          │
│               ↑ 外部Agent                              │
└────────────────────────────────────────────────────┘
                            │
                            ↓ spec 作为原料
┌─ proposal 产线 ─────────────────────────────────────┐
│ [追问机] → [proposal组装机] → [质检器] → [仓库]       │
│               ↑ 外部Agent    │                        │
│                              ↓ 违规                   │
│                         [回收站] → 修正提示词 → 重新入线│
└────────────────────────────────────────────────────┘
                            │
                            ↓ decision 作为原料
┌─ code 产线 ────────────────────────────────────────┐
│ [约束注入机] → [代码组装机] → [AST质检器] → [仓库]    │
│                  ↑ 外部Agent                           │
└────────────────────────────────────────────────────┘
                            │
                            ↓ 所有产出汇入
┌─ 审计产线 ─────────────────────────────────────────┐
│ [溯源机] → [trailer刻印机] → [归档]                   │
└────────────────────────────────────────────────────┘
```

### 2.2 共享基础设施

| 设施         | 状态来源             | Web 显示           |
| ------------ | -------------------- | ------------------ |
| 传送带流动 → | upstream 有效        | 绿色流动动画       |
| 传送带停滞 → | upstream 缺失        | 灰色 + 闪烁        |
| 电力面板     | 道/法元规则全在线    | 百分比 + 违规数    |
| 研究树       | 能力域 unlocked 状态 | 已解锁/锁定/解锁中 |
| 全局警报     | F/G/J 汇总           | 右上角感叹号       |

## 三、每台机器的交互

### 3.1 追问机

| 状态   | 外观     | 可操作                  |
| ------ | -------- | ----------------------- |
| 空闲   | 灰色     | 「开始追问」按钮        |
| 追问中 | 蓝色脉冲 | 追问表单（elicitation） |
| 完成   | 绿色     | 追问结果摘要            |

**点击「开始追问」** → 调用 `sihankor_propose` → 核心发送 `elicitation/create` → 浏览器渲染追问表单 → 用户回答 → 提示词生成。

### 3.2 组装机（外部 Agent）

| 状态     | 外观     | 可操作           |
| -------- | -------- | ---------------- |
| 等待原料 | 灰色     | 检查上游是否就绪 |
| 运行中   | 橙色旋转 | 「生成进度」     |
| 完成     | 绿色     | 产出链接         |

**不直接可点击。** 组装机本身不生成内容。内容由外部 Agent 生成——通过 `sampling/createMessage`。点击组装机显示"当前提示词"和"上次调用时间"。

### 3.3 质检器

| 状态     | 外观 | 可操作                  |
| -------- | ---- | ----------------------- |
| 未运行   | 灰色 | 「质检」按钮            |
| 运行中   | 蓝色 | 进度条                  |
| 通过     | 绿色 | 通过报告                |
| G 级违规 | 黄色 | 违规列表                |
| F 级违规 | 红色 | 违规列表 + 「修正」按钮 |

**点击「质检」** → `POST /api/actions/validate` → validator 运行 → WebSocket 推回结果。

**点击「修正」** → 生成修正提示词 → `sampling/createMessage` → 外部 Agent 修正 → 再验证。

### 3.4 回收站

| 状态     | 外观                | 可操作                  |
| -------- | ------------------- | ----------------------- |
| 空       | 灰色箱子            | 无                      |
| 有待修项 | 黄色箱子 + 数量角标 | 展开列表 + 「修正」按钮 |

**点击「修正」** → 对每份待修文档触发修正提示词。

### 3.5 仓库

只读。显示已入库文档的快照：id、stage、通过时间。

### 3.6 分拣器（stage 关卡）

传送带上的关卡，stage 推进卡片。

**点击** → 弹窗：确认从 2/3 推进到 3/3？→ 确认后 `POST /api/actions/stage`。

## 四、交互链路

### 4.1 浏览器 → 司衡

```text
浏览器按钮点击 → fetch() → POST /api/actions/* → Core 执行
                                                  ↓
                                         事件总线 → WebSocket 推送结果
                                                  ↓
                                         浏览器更新机器状态
```

### 4.2 Agent → 司衡 → 浏览器同步

```text
外部 Agent 调用 MCP tool → Core 执行 → 事件总线 → WebSocket → 浏览器更新
```

### 4.3 浏览器 → 司衡 → Agent 同步

```text
浏览器操作 → Core 执行 → 事件总线 → MCP 通知 → Agent 收到更新
```

Agent 不需要轮询。浏览器不需要轮询。事件总线是唯一同步源。

## 五、MCP 协议细节

### 5.1 追问链路

`elicitation/create` 标准协议。所有声明了 `elicitation` capability 的 host 都可渲染追问表单。

| host                       | 追问表单样式             |
| -------------------------- | ------------------------ |
| Reasonix                   | 选项按钮 + 文本输入      |
| Trae                       | 原生表单 UI              |
| Claude Code                | 对话内嵌卡片             |
| 不支持 elicitation 的 host | 降级：追问作为 text 返回 |

### 5.2 生成调度链路

`sampling/createMessage` 标准协议。司衡不持 API key——调用宿主 LLM。

| host            | `sampling` | 调度方式                                   |
| --------------- | :--------: | ------------------------------------------ |
| Claude Code     |     OK     | 全自动闭环                                 |
| Trae            |     OK     | 全自动闭环                                 |
| 不支持 sampling |     X      | 降级：提示词作为 text 返回，Agent 自主执行 |

### 5.3 降级模式

当 host 不支持 elicitation 或 sampling 时，司衡 tool 返回中直接嵌入结构化的文本提示词。Agent 读到后自主执行追问/生成，然后人把结果通过下一次 tool call 传回。

## 六、产线快照（image 降级）

非 Web 端用户通过 MCP `image` 类型接收产线状态：

- 服务端用 `resvg` 将 SVG 产线图渲染为 PNG
- tool 返回 `{type: "image", data: "base64...", mimeType: "image/png"}`
- 所有 Agent 都支持 `image` 内容类型

快照更新频率 30 秒（或事件驱动）。

## 七、端点设计

| 方法 | 路径                    | 说明          |
| ---- | ----------------------- | ------------- |
| GET  | `/`                     | 产线看板 HTML |
| GET  | `/api/dashboard`        | 产线状态 JSON |
| POST | `/api/actions/validate` | 触发验证      |
| POST | `/api/actions/stage`    | 推进 stage    |
| POST | `/api/actions/ratify`   | 签署 decision |
| POST | `/api/actions/correct`  | 触发修正      |
| POST | `/api/actions/grill`    | 启动追问      |
| WS   | `/ws/events`            | 实时事件推送  |

## 八、安全约束

| 操作       |     确认弹窗      | 不可逆 |
| ---------- | :---------------: | :----: |
| 追问       |        否         |   —    |
| 验证       |        否         |   —    |
| 推进 stage |      **是**       | **是** |
| 签署       | **是** + 输入签名 | **是** |
| 修正       |      **是**       |   否   |

推进和签署有二次确认。stage 不可逆转。

## 九、分期

| 阶段 | 交付                                 | 体量                  |
| ---- | ------------------------------------ | --------------------- |
| P0   | 只读产线看板：SVG 静态图 + 状态 JSON | Rust ~200 行 + 1 HTML |
| P1   | 按钮可操作 + WebSocket 实时推送      | Rust ~300 行 + JS     |
| P2   | 传送带动画 + 四条产线并行 + 电力面板 | 前端 ~400 行          |
| P3   | 研究树 + 能力域解锁动画              | 前端 ~300 行          |

## 十、@limitations

1. **产线快照是 30 秒刷新**，不是实时。实时只在 Web 端。
2. **降级模式体验受损**——host 不支持 sampling 时无法全自动闭环。
3. **分拣器推进不可逆**——设计如此，不是限制。
4. **所有按钮操作最终落在 Core**——浏览器是 Core 的一个窗口，不是独立系统。
5. **研究树的长远规划（P3）依赖能力域的实际代码就绪。**
