# 追问引擎采纳决策

## 背景

司衡当前覆盖治理链的中游（文档验证）和下游（AST 对齐），但上游缺失：用户意图在变成 proposal 之前，没有机制帮助用户精确化自己的意图。追问引擎提案（stage 2/3）设计了元规则驱动的四问题追问模型 + 约束提示词生成。

提案经审阅，方案完整。现推进至 decision 3/3。

## 方案选择

| 维度 | 决策 | 法依据 |
|------|------|--------|
| 追问模型 | **采纳**：四问题一次性追问（道二/顺因/有度/知止） | 顺因（因果链由上游元规则驱动） |
| 提示词生成 | **采纳**：frontmatter 模板 + 章节结构 + 可证伪条件 | 有度（生成边界由追问结果限定） |
| 修正闭环 | **采纳**：复用已有 validator，不新增修正逻辑 | 知止（不重复造轮子） |
| Solo 模式追问 | **采纳**：一次性追问，不采用轮询式 | 有度（工程师不需要被反复打断） |
| Code 模式追问 | **不在本次范围** | 知止（教育层追问留到后续迭代） |

## 备选方案

| 方案 | 描述 | 不采纳理由 |
|------|------|-----------|
| A：Pocock 风格轮询式 | 一次一个问题，逐题追问 | Solo 模式的工程师需要快速推进，不适合反复打断 |
| B：追问引擎自调 LLM | 追问引擎持有 API key，自己调 LLM 生成内容 | 违反核心定位：司衡不生成代码/文档，只验证。而且验证者与生成者做在同一进程内，鉴的公信力减弱 |
| **C：采纳（本次）** | MCP tool `sihankor_propose` 返回提示词，由外部 Agent 消费 | （见理由） |

## ADR

### decided-by

moc

### 理由

1. **顺因**：追问引擎是 Mind 上游扩展，方向正确——意图在 iCL 之前收敛，因果链从元规则→追问→提示词→外部生成→验证，方向单一可追溯。
2. **有度**：`src/mind/grilling.rs` 约 500 LOC，模块独立，不修改 Mind 三机流转。四条追问与元规则一一映射，无冗余。
3. **知止**：明确不做什么——不轮询追问、不自调 LLM、不做 Code 模式教育追问、不做 domain-modeling 词汇管理。边界清晰。
4. **损补**：补足司衡治理链的上游空白——意图→proposal 阶段。当前此间隙由人自行弥合，追问引擎提供结构化框架减少弥合成本。

### 后果

#### 正面后果

| 后果 | 说明 |
|------|------|
| 上游间隙缩小 | 用户意图→proposal 之间有了治理驱动的收敛工具 |
| 验证器规则可注入提示词 | 追问结果可前置 validator 的已知约束（如禁止 → 字符），减少生成后再修正的次数 |
| MCP 交互面扩展 | `sihankor_propose` 成为司衡的第一个主动调度接口 |
| 自指验证 | 本决策的生成验证了追问引擎的必要性——如果本决策的 proposal 阶段使用了追问引擎，"siheng 旧名仍在 prompt 中使用"的 bug 可被顺因追问（"这个名字对吗"）提前发现 |

#### 代价与风险

| 代价 | 说明 |
|------|------|
| 追问质量依赖问题设计 | 如果四个元规则问题流于形式（被用户机械回答），追问引擎产出不优于无追问 |
| 验证器规则未编码时不生效 | 追问引擎不能前置注入验证器还不知道的规则——CorrectionTracker 未实现前，新违规模式仍只能事后发现 |
| 外部 Agent 不遵守提示词 | 约束注入的有效性取决于外部 Agent 是否遵守结构约束——司衡无法强制 Agent 的生成行为 |

### 可证伪条件

| 条件 | 证伪方法 | 时间窗口 |
|------|---------|---------|
| 追问引擎减少上游间隙 | 统计使用前后 proposal 的 validator 首次违规数对比 | 追问引擎上线后 6 个月 |
| 约束注入降低修正循环次数 | 统计修正闭环触发次数 / proposal 生成总数 | 追问引擎上线后 3 个月 |
| 追问不显著增加用户负担 | 提问数 4，用户回答时间 < 5 分钟（通过对超过 5 分钟的会话做抽样分析） | 追问引擎上线后 3 个月 |
