# DSR-3 对司衡定位的影响

> DSR-3（bat + fd 外部基线采集）暴露了"治理运行时"定位与当前实现之间的结构性矛盾。本文记录四个盲区及其战略含义。

## 一、DSR-3 的独特价值

DSR-1/2 在 SiHankor-A（司衡近亲）上运行，验证了治理运行时在同源项目上的全链路能力。DSR-3 首次在两个不共享司衡 DNA 的开源项目（bat、fd，作者 sharkdp）上并行部署。

DSR-3 不是又一个数据点——它的两组基线（bat：主动升级 .md 到 .sih.md，0→2 索引；fd：仅修复格式问题，0→0 索引）共同暴露了四个结构性盲区。

## 二、四个盲区

### 2.1 硬扩展名：司衡的入口是 .sih.md

`discover_documents` 硬编码 `name.ends_with(".sih.md")`。bat 有 14 个 `.md` 文件，fd 有若干——全部被跳过。没有 `.sih.md` 就没有索引、没有度量、没有行迹。

**含义**：当前司衡不是"运行时"，是"笔记本"——它只能读自己写的格式。一个真正的治理运行时应该能观察陌生系统的文档结构，生成治理评估报告，而不是要求系统先格式化自己。DSR-3 bat 证明了代理可以手动创建 `.sih.md` 来"跨进治理门"，但外部项目的入口不应该依赖代理的主动干预。

### 2.2 DB 隔离：跨项目部署的前提条件

`find_db_path` 不读 `SIHANKOR_DB_PATH` 环境变量。切换 docs_dir 时，度量管道返回旧项目的累积数据。DSR-3 bat 中修复了 `SIHANKOR_DB_PATH` 支持，DSR-3 fd 没设这个变量——所有度量数字是司衡全局 DB 的视图，不是 fd 自身。

**含义**：这是工程实现问题，已被修复。但它暴露了一个更深的问题：司衡假设一个二进制只服务一个文档树（它自己）。如果定位是"治理运行时"，多项目隔离是基础设施而非可选特性。

### 2.3 极端度量：一刀切规则对零治理项目的扭曲

DSR-3 bat 的规则密度为 7.00（2 个文档承受 14 条规则），DSR-2 的 SiHankor-A 为 0.10。这不是规则错了——14 条规则是合理的司衡内部治理标准——但向外投射到零治理项目时，一刀切的规则集产生了不匹配的治理力度。

**含义**：有度（G3，比例原则）要求"治理力度与风险成正比"。如果司衡面向非治理项目，它的规则力度应该随项目的治理成熟度而调整——一个连 `.sih.md` 都没有的项目不应该承受和司衡自身相同密度的规则集。这不只是"先 warn 不强断"的问题，是度量数字本身的解释框架需要按治理成熟度分级。

### 2.4 治理策略决定结果，不是司衡能力

bat 和 fd 同一作者、同样 Rust、同样目录结构。唯一差异是代理的治理策略选择——bat 升级 `.md` 到 `.sih.md`，fd 不升级。bat 有 2 个索引、2 行迹、可观测的治理效果；fd 有 0 索引、0 度量的治理效果。

**含义**：司衡对非治理项目的价值完全取决于"是否有人愿意投入治理成本"。这和"治理运行时"的定位矛盾——运行时应对治理间隙产生观测，而不是被动等待被治理对象格式化。

## 三、战略含义

### 3.1 从"笔记本体"到"治理运行时"的关键一跳

如果司衡真正想成为"面向 AI 编码的治理运行时"，下一阶段最重要的不是强化现有的 14 条规则集，也不是继续在近亲项目上堆 DSR 周期数，而是**打通非治理项目的观测路径**。

具体方向：让司衡不依赖 `.sih.md` 扩展名就能对陌生项目生成治理基线报告——发现文档结构、统计 Markdown 合规度、评估缺失的治理元素（如 ADR、stage 标记、upstream 链）。不为强制合规，为生成"治理成熟度快照"。

### 3.2 这不是放弃 .sih.md

`.sih.md` 作为司衡自身的治理格式没有问题。问题在于把它作为**所有被治理项目的入口条件**。入口应该是项目路径，不是格式转化。

### 3.3 DSR 数据的诚实含义

三个 DSR 周期的数据汇总：

| DSR | 项目类型 | 索引率 | 规则密度 |
|-----|---------|--------|---------|
| DSR-2 | 司衡近亲 | 43% | 0.10 |
| DSR-3 bat | 外部（主动进入） | 0→2 干预 | 7.00 |
| DSR-3 fd | 外部（留在门外） | 0 | N/A |

趋势很清楚：离司衡 DNA 越远，治理覆盖越低。这不是失败——是诚实的数据。诚实的数据驱动诚实的决策。当前的前进方向不是"如何让外部项目更接近司衡"，而是"司衡如何更接近外部项目"。

## 四、后续建议

1. 将"非治理项目基线观测"作为 DSR 的一个固定阶段——不是替代现有的治理迭代，是新增一个观测模式
2. 探索 Markdown 通用解析路径——让 indexer 能发现 `.md` 文件并生成治理成熟度评分，不要求 `.sih.md` 扩展名
3. 在度量管道中增加"治理成熟度"维度——对零治理项目的极端值不再是 bug，而是分级基线
