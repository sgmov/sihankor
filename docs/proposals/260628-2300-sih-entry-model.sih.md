# 司衡双入口模型

> DSR-3 暴露了 `.sih.md` 硬入口对外部项目的屏障。本文定义司衡的双入口模型——治理门和观测窗——取代当前的单一入口。

## 一、问题

DSR-3 bat/fd 暴露了一个结构性矛盾：司衡定位为"治理运行时"，但入口硬编码 `.sih.md` 扩展名。没有 `.sih.md` 的项目完全不可见——索引为 0、度量为 0、行迹为 0。

如果要求所有被治理项目先学会司衡的治理语言才能被观测，司衡就不是运行时，是笔记本。真正的运行时应该能观察陌生系统，生成治理评估报告，让用户决定是否进入严格治理。

## 二、双入口模型

| | 治理门 | 观测窗 |
|---|---|---|
| 入口条件 | `.sih.md` + frontmatter | 项目路径（`.md` 文件） |
| 做什么 | 验证合规、阻断违规 | 扫描文档结构、统计合规度 |
| 改写文件 | 是（修复后重新索引） | 否（只观测不改写） |
| 目标 | 治理合规 | 治理可观测 |
| 度量管道 | 完整 | 仅结构级（无 violation 级） |
| 行迹 | 支持 | 不支持（观测窗只读） |

用户路径：

```
项目 → 观测窗（扫描 → 报告）
                ↓
         用户决策
         ↙      ↘
   治理门      不进门
 (.sih.md     （报告即
  + 合规）     最终交付）
```

## 三、sih-docs/ 目录

对于愿意引入司衡治理但无法改造 `docs/` 的项目，司衡支持 `sih-docs/` 作为独立的治理文档根：

```
项目/
  docs/           ← 项目自己的文档，格式不变，司衡不碰
  sih-docs/       ← 司衡治理文档，独立存在
    specs/
    proposals/
    decisions/
    knowledge/
      trails/
```

三种场景全覆盖：

| 场景 | docs_dir 配置 | 说明 |
|------|-------------|------|
| 全量司衡用户 | `"sih-docs"` | `sih-docs/` 是唯一文档根，项目没有 `docs/` |
| 有自己 docs 的用户 | `"sih-docs"` | 司衡只看 `sih-docs/`，`docs/` 不受影响 |
| 纯观测用户 | `"docs"` | 扫描但不做合规判断，不要求 `.sih.md` |

SiHankor-A 已用 `sih-docs/` 模式验证了可行性。

## 四、docs/ 与 sih-docs/ 的关系

两个目录回答不同的问题，不存在冲突：

| 问题 | 看哪里 | 为什么 |
|------|--------|--------|
| "这个模块怎么用？" | `docs/` | 实现细节 |
| "为什么选了这个架构？" | `sih-docs/decisions/` | ADR 有阶段、上游链、决策理由 |
| "这个变更被批准了吗？" | `sih-docs/proposals/` | stage 3/3 = 已决议 |
| "现在系统实际怎么跑的？" | `docs/` + 源码 | 当前状态 |

当 `docs/` 说 X 而 `sih-docs/specs/` 说 Y 时，这不是冲突——这是**治理间隙**，是道四（"规约与实现必有间隙"）在双目录模型中的直接体现。司衡的价值不在两者间选一个，在让间隙可见。

Agent 决策规则：`sih-docs/` 有治理权威（stage、upstream、decided-by），`docs/` 有实现权威（当前代码状态）。不一致时两面都读，以 `sih-docs/specs/` 为意图锚点、以 `docs/` + 源码为当前事实。

## 五、三步工程路径

### 5.1 第一步：纯观测窗（当前可做）

让 `discover_documents` 在 `--include-md` 标志下扫描 `.md` 文件——不检查 frontmatter、不执行 F/G/J 规则、不阻断。产出一份结构化观测报告：

- 文档数量、目录结构
- 表格合规度（几列）
- 代码块语言标签覆盖率
- 如果引入司衡治理，需要多少改造

当前 `PipelineConfig` 已支持 `SIHANKOR_DOCS_DIR` 环境变量。观测窗在此基础上扩展 `--include-md` 标志。

### 5.2 第二步：映射适配（中等工程）

用户可选提供 `sihankor-map.yaml`，声明项目自有 frontmatter 与司衡字段的等价关系。司衡读取后执行合规判断，但不改写原文件。报告会标注"合规（通过映射）"与"需修复"的区别。

### 5.3 第三步：治理门（已有）

当前 `.sih.md` 治理门保持不变。三步中只有第三步会改写文件。

## 六、@limitations

1. **观测窗不解决语义冲突**：`docs/` 和 `sih-docs/` 的不一致由人工识别和协调，司衡只做表面扫描不做语义分析
2. **映射适配依赖用户正确声明等价关系**：如果用户声明 `status: done` 等价于 `stage: 3/3`，但实际文档质量达不到 ratify 标准，司衡不会感知到
3. **sih-docs/ 是独立治理空间**：它不是 `docs/` 的映射层，不会自动同步。需要人工维护两个目录的对应关系
