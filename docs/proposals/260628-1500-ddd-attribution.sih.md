# DDD 归属提案

## 一、问题

### 1.1 DDD 在司衡构建中被反复提及

DDD（Domain-Driven Design）在司衡引擎构建过程中以显式和隐式方式被反复提及：

哲学论证集开放研究方向 #2 提出 DDD 归属问题，初步判断"洞察归法层，实践归术层"，但标注为"新体系文档未改变此状态"，悬而未决。

Clean Architecture 证伪笔记使用 DDD 领域事件作为反例挑战分层依赖假设。

Conway's Law 证伪笔记提出需要对 DDD Bounded Context 执行九段式检验以增加样本量。

### 1.2 反复提及不是偶然

DDD 的核心概念与司衡的结构存在深层对应：

| DDD 概念 | 司衡对应 | 对应层级 |
| ---- | ---- | ---- |
| 限界上下文 | 文档目录治理的语义边界 | 法层（顺因：意图边界先于实现边界） |
| 聚合根 | 治理力度的收敛点 | 法层（知止：治理力度应有边界） |
| 领域事件 | 三机流转的事件驱动 | 机制层（三机） |
| 通用语言 | 术语血统表与 glossary | 法层（顺因：规范先于实现） |

这不是表面相似，是结构对应。司衡的五法可以在 DDD 的战略设计中找到结构原型。

## 二、归属判断

### 2.1 DDD 的核心洞察已融入五法

DDD 的核心洞察不是术，是法层的观察来源。限界上下文揭示了"语义边界"的存在，对应顺因（意图边界先于实现边界）。聚合根揭示了"结构收敛点"的存在，对应知止（治理力度应有边界）。这些洞察已经被五法吸收，只是未显式标注来源。

将 DDD 的核心洞察归为法层的观察来源，不是将 DDD 整体纳入法层。五法的推导链不依赖 DDD，DDD 只是五法在软件工程传统中的结构性印证。

### 2.2 DDD 的战术设计是候选术

DDD 的战术设计（聚合、实体、值对象、领域事件、仓储等模式）是 spec-coding 之外另一种将意图显式化的工程化展开。spec-coding 从文档侧显式化意图，DDD 从代码结构侧显式化意图，两者覆盖意图生命周期的不同维度。

### 2.3 不立即纳入术层的理由

直接将 DDD 纳入术层存在三个问题：

顺因问题：加入 DDD 需要先论证 DDD 与五法的映射关系，否则是术先于法。

鉴的暂缓：DDD 尚未经九段式检验，而九段式当前暂缓。在检验完成前将 DDD 纳入术层，等于将未经验证的实践固化为治理机制。

知止问题：当前 spec-coding 尚未完全落地（链 2 代码侧 L5），过早纳入第二术可能分散治理资源。

## 三、处理方式

### 3.1 显式声明归属

将哲学论证集开放研究方向 #2 的 DDD 归属判断从"初步判断"升级为明确声明：

DDD 的核心洞察已融入五法（法层观察来源），DDD 的战术设计是 spec-coding 之外的候选术，待九段式检验后决定是否纳入。

### 3.2 不立即修改机制层 5.1

机制层 5.1 当前只授权 spec-coding 和三机。DDD 的战术设计不立即纳入，待以下条件满足后再考虑：

- 九段式完成暂缓处置（实现或退役）
- DDD 通过九段式检验
- spec-coding 在工程映射中的 L 级别提升到 L2 以上

### 3.3 工程映射新增候选链

新增链 16：道二/道三 -> DDD 战术设计。初始 L 级别 L5（零实现 + 待检验）。

| 项目 | 内容 |
| ---- | ---- |
| L 级别 | L5 |
| 哲学要求 | 代码结构侧的意图显式化 |
| 工程现状 | 零实现，待九段式检验 |
