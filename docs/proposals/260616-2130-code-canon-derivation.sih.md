# 代码术层约束的法层推导

> 本文档从四道推导代码域的法层规则（L-14 起），供 Canon Reopen 时并入 $3.8 生命周期规则表。
> 当前为草稿（1/3），需经鉴审阅后晋升 2/3，Canon 正式并入后晋升 3/3。

## 一、正名：为何需要代码域的法层规则

Canon 现有 L-01~L-13 全部治理文档生命周期。代码从未进入法层视野。结果是：术层约束（如「禁止 unwrap」）的合法性仅基于工程惯例，无法追溯至道。

道三「代码自晦，意图必复」是对代码的断言。如果这是道，它必须在法层产生对应的约束——否则道就只是修辞。

法不定义「代码具体怎么写」（那是术），法定义「术层代码约束如何合法建立」以及「哪些代码属性必须被术层覆盖」。

## 二、顺因：道→法推导链

### 道一「收敛必为」→ 代码法

**道的断言**：任何发散必须最终收敛。不收敛的发散是熵增。

**代码域的映射**：代码如果偏离规约而不自知，就是发散。代码必须可追溯至已被 ratify 的规约。

### 道二「因果方向」→ 代码法

**道的断言**：意图先于实现。因果方向不可逆。

**代码域的映射**：API 的公开签名是意图的声明。如果签名没有文档——即没有表达意图——那么使用者只能从实现逆向猜测意图。这颠倒了因果方向。

### 道三「代码自晦，意图必复」→ 代码法

**道的断言**：代码不透明。必须通过明确的约束保证意图可被复原。

**代码域的映射**：命名、结构、文档必须服务于「意图复原」这一目标。如果一个符号的名字不能表达它的意图，法要求修正它。

### 道四「规约与实现必有间隙」→ 代码法

**道的断言**：间隙不可消除，只能暴露和管理。

**代码域的映射**：
- `unwrap()` 是将间隙暴露为运行时 panic 的机制——它没有管理间隙，而是把间隙变成了灾难。
- `dbg!`/`println!` 是调试过程中的临时桥梁——它承认间隙的存在（调试期间），但不应该残留在 ratify 代码中（那是将临时桥梁当作了永久建筑）。
- unreachable 代码——不可达路径——应该是类型系统消除的，而不是人类声称的。

### 鉴「诗理区分」→ 代码法

**鉴的断言**：可证伪的约束是理，不可证伪的约束是诗。

**代码域的映射**：术层约束在被设为 deny 之前，必须经过鉴的检验。未经鉴的约束不得阻碍编译。

## 三、法层规则

| 编号 | 规则 | 类型 | 道层依据 | 法层依据 |
|------|------|------|----------|----------|
| L-14 | 代码须可追溯至 ratify 规约 | 引用 | 道一（发散须收敛） | 顺因 |
| L-15 | 公开 API 须声明意图（doc comment） | 术层准入 | 道二（因果方向）+ 道三（意图必复） | 顺因 |
| L-16 | 错误路径须暴露间隙，不得隐藏为 panic | 术层准入 | 道四（间隙必暴露） | 有度 |
| L-17 | 调试痕迹不得残留在 ratify 代码中 | 术层准入 | 道四（临时桥梁非永久建筑） | 知止 |
| L-18 | 不可达路径须从类型层面消除，非人类声称 | 术层准入 | 道三（代码自晦——人类的「不可能」往往是错的） | 知止 |
| L-19 | 术层 deny 约束须经鉴检验方可生效 | 术层准入 | 鉴（诗理区分） | 有度 |

### L-14：代码须可追溯至 ratify 规约

**释义**：每一段代码的存在理由必须可追溯至一份 ratify（3/3）的 spec 文档。这不是要求在每行代码上加注释指向文档 ID，而是要求引擎能回答「这段代码实现了哪个规约的哪条要求」。

**违例**：存在一段代码，无法在 docs/specs/engineering/ 中找到对应的规约声明。

**法层依据（顺因）**：道一要求发散收敛——如果不收敛于规约，代码的存在理由就是发散的。顺因要求因果链完整——规约是代码的因。

### L-15：公开 API 须声明意图

**释义**：所有 `pub` 项（函数、类型、trait、模块）必须有 doc comment（`///`）。doc comment 的内容是意图声明，不是实现描述——它回答「是什么、为什么存在」，不是「怎么工作的」。

**违例**：一个 `pub fn` 没有 doc comment，或 doc comment 仅描述实现细节而无意图声明。

**法层依据（顺因+道三）**：道二要求意图先于实现——doc comment 是意图的文本表达，必须在代码之前存在（至少同时）。道三要求意图可复原——没有 doc comment，使用者只能逆向猜测，意图不可复原。

### L-16：错误路径须暴露间隙

**释义**：`unwrap()` 和 `expect()` 是间隙的隐蔽处理——它将规约与实现之间的间隙（Value 可能是 None）暴露为运行时 panic。正确的方式是使用 `Result` 传播，或将不可恢复的错误转换为明确的 panic 消息并记录理由。

**违例**：非测试代码中使用 `unwrap()` 或 `expect()`。

**法层依据（有度）**：道四要求间隙被管理——unwrap 没有管理间隙，它是把间隙隐藏起来让运行时去承受。有度要求工具的边界清晰——unwrap 的边界是「人类声称这里不可能 None」，而道四的断言是「人类的声称不可靠」。

### L-17：调试痕迹不得残留

**释义**：`dbg!()`、`println!()`、`eprintln!()` 是调试过程中的临时桥梁——它们帮助人类在开发阶段观测间隙。ratify 代码中不应残留这类痕迹，因为 (a) 它们是噪声，(b) 它们表明规约尚未完全收敛于代码。

**违例**：非测试代码中残留 `dbg!()` 或裸 `println!()`。

**法层依据（知止）**：道四说间隙永远存在。但调试痕迹是对「当前正在发现的间隙」的临时标记。代码 ratify 后这些痕迹应该消失——要么间隙已被处理（代码已修正），要么间隙已被 formalize（变成了正式的日志/错误处理）。残留的 `dbg!` 是「知道有间隙但没处理也不想处理」的沉默承认。知止要求知道什么时候该停——调试在 ratify 时应该停止。

### L-18：不可达路径须类型消除

**释义**：`unreachable!()` 宏和 `todo!()` 宏是人类对「这段代码不会/尚未执行」的断言。但道三说代码自晦——人类的「不可能」经常是错的。真正的不可达应该通过类型系统表达（`enum` exhaustiveness、`NonZeroUsize` 等）。

**违例**：代码中使用 `unreachable!()` 或 `todo!()`。

**法层依据（知止）**：知止要求认识到人类的判断有边界。当人类声称「这段代码不可达」，实际上是在说「在我当前的认知模型中不可达」——而道四说认知模型必然有间隙。类型系统消除不可达路径是知止的实践：把判断从人类转移给编译器。

### L-19：deny 约束须经鉴

**释义**：任何设为 `deny` 级别的 lint 规则，必须能够说明：(1) 它从哪个道推导而来，(2) 它的违例如何被证伪——即在什么条件下这条规则应该被降级或移除，(3) 它经过了鉴的段三（最强反证）检验。

**违例**：一条 deny 级规则没有经过鉴的检验记录。

**法层依据（有度）**：鉴区分诗和理。如果把诗当作理来 deny，就是过度约束——它阻碍了代码的意图表达而非促进。有度要求每条约束都有边界——鉴就是确定边界的工具。

## 四、有度：L-14~L-19 的边界

**管**：
- 代码与规约的可溯源性（L-14）
- 公开接口的意图声明（L-15）
- 间隙的暴露机制（L-16/L-17/L-18）
- deny 约束的合法建立（L-19）

**不管**：
- 代码的具体实现方式（算法、数据结构）——那是术层
- 性能指标——那是 profiling 的域
- 测试策略——那是独立的 CI/测试域
- API 设计美学——那是 code review 的域

## 五、道四：解决的间隙

当前间隙：Canon 只治理文档，不治理代码。但 SiHankor 作为一个「治理体系」，其治理对象包含代码——引擎本身就是代码。如果一个治理体系只治理自身的文档而不治理自身的代码，这是道四的自我否定：体系声称管理间隙，却放任自己的代码处于法层真空。

L-14~L-19 填补这个间隙：代码从法层的真空进入治理视野。

## 六、鉴：九段反推法检验 L-14~L-19

以下对六条法层规则逐段过炉。

### 段一：主张提取

每一条 L 规则隐含一个主张——「这条约束是道的必然推论，而非术层的审美偏好」。六条主张：

| 规则 | 主张 |
|------|------|
| L-14 | 代码必须可追溯至规约——这是道一的必然推论 |
| L-15 | 公开 API 必须声明意图——这是道二+道三的必然推论 |
| L-16 | unwrap 不得隐藏间隙——这是道四的必然推论 |
| L-17 | 调试痕迹不得残留——这是道四的必然推论 |
| L-18 | 不可达路径须类型消除——这是道三+知止的必然推论 |
| L-19 | deny 约束须经鉴——这是鉴的方法论必然推论 |

### 段二：概念分析

| 概念 | 精确化定义 |
|------|-----------|
|「可追溯」| 代码单元（函数/模块/类型）的存在理由可被定位到一份 ratify spec 中的语句。不是每行代码，是每个语义单元 |
|「公开 API」| `pub` 项——但不包括 trait impl 中的方法（trait 自身已声明意图）和宏展开产物 |
|「间隙暴露」| 运行时故障必须是可恢复的（Result）或携带明确上下文的（带消息的 panic），不能是裸 unwrap 的「panicked at line 42」 |
|「调试痕迹」| `dbg!()` / 裸 `println!()` / `eprintln!()` ——不包括正式的日志框架（`log`/`tracing` crate） |
|「类型消除」| 不可达性由类型系统保证（enum 穷举、NonZero 类型等），而非人类判断 |
|「经鉴」 | 经九段反推法至少段一至段八，段八判定为 SURVIVES 或 CALIBRATE（非 FALSIFIED/UNFALSIFIABLE） |

### 段三：最强反证构建

**对整体框架的反证**：「Canon 是文档治理体系。将代码约束纳入 Canon，一是超出了 Canon 的设计域，二是代码约束的更合适载体是 clippy 配置而非哲学文献。」

这个反证很强——它不否认规则的价值，但质疑其位置。回应：道三对代码做了断言（「代码自晦」）。如果道的断言是真，法必须覆盖代码域——否则道的断言就是不可执行的修辞。Canon 当前 L-01~L-13 全部是文档域，这是因为 SiHankor 起源于文档治理，不是法层有意识地排除了代码。纳入代码约束不是扩展域，而是填补遗漏。

**对各规则的反证**：

| 规则 | 最强反证 |
|------|----------|
| L-14 | 「每行代码都要追溯 spec」不现实——工具函数、标准库封装、胶水代码没有 spec |
| L-15 | Rust 生态惯例允许 `pub fn new()` 无文档。强制 doc 是形式主义 |
| L-16 | `1u32.checked_add(2).unwrap()` 逻辑上不可能 panic——禁止它是噪音 |
| L-17 | 二进制程序的 main 函数中 `println!("Starting...")` 是操作性输出，不是调试痕迹 |
| L-18 | `todo!()` 是开发阶段的合法标记，不应在 1/3 阶段被 deny。`unreachable!()` 在穷举 match 中是合理的安全网 |
| L-19 | 鉴要求自身也经检验（$8.2 自承不完备）。L-19 作为一种「自身也经鉴」的规则，是否产生无限递归？ |

### 段四：反例举证

**L-14 反例**：`src/core/database.rs` 中的 `row_to_document` 是一个内部转换函数——它没有对应的 spec 语句，但它的存在理由（从 SQL 行转换为 Document 结构体）是显而易见的。若 L-14 要求「每个函数必须指向 spec 语句」，这是过度约束。

**L-15 反例**：`impl Default for ValidationConfig` —— Default trait 的契约在 std 中已文档化。为这个 impl 加 `/// Returns the default validation config` 是冗余。

**L-16 反例**：`usize::from_str("42").unwrap()` 在常量解析场景中不可能失败——unwrap 只是告诉编译器「这不会失败」，而非隐藏间隙。

**L-18 反例**：Rust 编译器在穷举 match 中允许 `unreachable!()` 作为 safety net——当未来新增 enum variant 时，编译器会报错而非静默 panic。这种用法是在类型层面处理不可达。

### 段五：类比检验

其他治理体系或工程规范如何处理代码约束：

| 体系 | 代码规则 | 法层依据 |
|------|----------|----------|
| MISRA C | 禁止 malloc、禁止递归 | 安全关键系统的形式化需求 |
| Rust API Guidelines | pub 项必须文档化 | 生态惯例，非形式化推导 |
| Google C++ Style Guide | 禁止异常、要求 const | 工程经验累积 |

共性与差异：所有成熟工程规范都有代码约束，但它们的合法性来源不同——MISRA 来自安全分析，Rust 惯例来自社区共识，Google 指南来自经验。SiHankor 的独特之处：从哲学的道推导法层约束，而非从经验倒推。这是更强的合法性来源，但如果推导不严谨，也是更危险的自欺。

### 段六：逻辑一致性

检验 L-14~L-19 与现有 SiHankor 体系的内部一致性：

| 检验项 | 判定 |
|--------|------|
| L-14 是否与道一一致 | ✅ 收敛必为 → 代码必须收敛于规约 |
| L-15 是否与道二/道三一致 | ✅ 意图先于 + 意图必复 → API 声明意图 |
| L-16 是否与道四一致 | ✅ 间隙必管理 → unwrap 是间隙隐藏非管理 |
| L-17 是否与道四一致 | ✅ 临时标记非永久建筑 → dbg! 是开发期桥梁 |
| L-18 是否与道三/知止一致 | ✅ 人类判断有边界 → 类型消除比人类声称更强 |
| L-19 是否与鉴的自我参照一致 | ✅ 鉴声言自身不完备 → L-19 是对此的方法论执行 |
| Canon 的文档聚焦是否被破坏 | ⚠️ 需在 Canon $四补充「法层覆盖代码」声明 |
| 与 Dev-Governance 是否冲突 | ✅ Dev-Governance 已预留代码治理域 |

### 段七：可证伪条件

| 规则 | 证伪条件 |
|------|----------|
| L-14 | 找到一类代码，其存在理由不可追溯到任何 spec，但 (a) 删除它会破坏系统，(b) 无法通过重构消除 |
| L-15 | 找到一个 pub API，其意图从签名和类型完整可推断，且 doc comment 只会增加维护负担 |
| L-16 | 找到一处 unwrap，其中 None 在类型层面被证明不可能（而非人类声称不可能） |
| L-17 | 找到一处 println!，它不是开发期调试痕迹而是程序的操作性输出（如 CLI 的用户界面） |
| L-18 | 找到一处 unreachable!，其不可达性由类型系统自动保证（如 exhaustive match + safety net） |
| L-19 | 找到一条 deny 规则，其损伤（阻碍合法编码）超过收益（防止真实缺陷），且 ratio 可量化 |

### 段八：证伪判定

| 规则 | 判定 | 理由 |
|------|------|------|
| L-14 | **CALIBRATE** | 概念分析中的「语义单元」需要精确定义。段四反例成立——内部转换函数不需要 trace to spec。校准：L-14 的范围收缩为「公开 API 和领域逻辑」，排除内部胶水代码 |
| L-15 | **CALIBRATE** | 段四反例成立——trait impl 中 trait 已声明契约。校准：L-15 排除 trait impl（trait 自身的 doc 已覆盖） |
| L-16 | **CALIBRATE** | 段四反例部分成立——常量解析中的 unwrap 是形式上的而非实质的间隙隐藏。校准：L-16 区分「逻辑上可能的间隙」和「类型上已消除的间隙」。后者允许 `expect("by construction")` |
| L-17 | **CALIBRATE** | 段四反例成立——程序的操作性输出不是调试痕迹。校准：L-17 排除二进制程序 main 函数中的 println!/eprintln! |
| L-18 | **CALIBRATE** | 段四反例成立——`unreachable!()` 在 exhaustive match 中是安全网。`todo!()` 是开发标记不应 deny。校准：L-18 `unreachable!` 降为 warn；`todo!` 降为 warn |
| L-19 | **SURVIVES** | 鉴的自我参照是特性而非缺陷。L-19 的方法论要求（段一至段八，SURVIVES 或 CALIBRATE）本身已满足自身要求 |

### 段九：校准建议

基于段八的判定，六条规则全部需要 CALIBRATE 后采纳。校准后的规则表：

| 编号 | 校准后规则 | 校准内容 |
|------|-----------|----------|
| L-14 | 领域逻辑和公开 API 须可追溯至 ratify 规约；内部胶水代码豁免 | 收缩范围 |
| L-15 | pub 项须声明意图（doc comment）；trait impl 中 trait 已声明契约的豁免 | 排除 trait impl |
| L-16 | 间隙不得隐藏为裸 panic；逻辑上已证明安全的 expect 须加 `// by construction` 注 | 区分逻辑间隙与类型消除 |
| L-17 | 调试痕迹不得残留；二进制程序的操作性 I/O 豁免 | 区分调试与操作 |
| L-18 | 不可达路径优先类型消除；exhaustive match 中的 unreachable 安全网允许；todo! 仅开发期可接受 | 降级+允许例外 |
| L-19 | deny 约束须经鉴（段一至段八，SURVIVES/CALIBRATE）| 维持原样 |

## 七、与 code-lint 提案的关系

code-lint 提案（`260616-2100-code-lint-proposal`）的术层规则（8 deny + 12 warn）需要法层授权。在 L-14~L-19 进入 Canon 之前，code-lint 提案的 deny 规则的合法性是悬空的。

建议顺序：
1. 本文档审阅通过 → 晋升 2/3
2. Canon Reopen → 正式并入 L-14~L-19
3. code-lint 提案的 upstream 改为 Canon（而非 Assay）+ 按 L-19 补充鉴检验记录
