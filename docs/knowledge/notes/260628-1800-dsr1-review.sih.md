## 五、发现与洞察

### 5.1 DSR 流程可行

最小六步流程（部署→索引→基线→迭代→后快照→回顾）在两个小时内完成。司衡的索引管道在非司衡项目上运行正常，`infer_nature` 对 SiHankor-A 的目录结构（`sih-docs/` → 临时映射为 `docs/`）能部分工作。

### 5.2 Stage 格式是跨治理系统的最大间隙

SiHankor-A 的文档体系有自己成熟的治理想法（`propose`/`resolve`/`ratify` 阶段编码），与司衡的 `1/3`/`2/3`/`3/3` 格式是命名约定的差异，不是治理能力的差异。38 个错误中的 25 个属于这种情况。这验证了道二（编码必有损）和道四（规约与实现必有间隙）：即使两个司衡系统共享"治理"概念，工程实现的细节差异仍然产生间隙。

### 5.3 Archive 目录暴露治理边界问题

所有 stage 格式错误集中在 `archive/`。这揭示了 `infer_nature` 的一个缺口：archive 目录没有对应的 nature 映射（当前映射为 `None`，nature 推断失败）。未来应考虑为 archive 增加专门的 nature 处理。

### 5.4 度量管道未完全激活

本次 DSR 仅使用了索引管道（`rebuild_index`），度量管道（variance_metric/snapshot_diff/trend_alignment）因需要 MCP server 交互而未采集。完整度量需要：
- 启动 MCP server 并配置 docs_dir 指向目标项目
- 调用 `project_status` 采集 `ProjectSnapshot`
- 调用 `variance_metric`/`snapshot_diff` 产出对比报告

## 六、DSR-2 建议

1. **启用度量管道**：在 DSR-2 中启动 MCP server，采集完整的 `ProjectSnapshot` + `VarianceMetric`
2. **扩大修复范围**：修复 archive/ 的 stage 格式，观察治理投入与产出的关系
3. **多 agent 场景**：利用 SiHankor-A 的 zai 代理 propose 文档，观察多 agent 治理数据
4. **目录映射**：考虑在索引配置中支持自定义 nature 映射，消除 `sih-docs/` → `docs/` 的临时映射
