# 多轮独立检验治理模式：蓝皮书融入案例的元洞察

> 2026-06-17~18，一份外部 AI 生成的司衡进化蓝皮书，经三轮反推检验、三轮独立对照（每轮两份独立检验 + 一份对比裁决）、一次校准提案、一次决策记录，完成了从 `knowledge/drafts/` 到 proposal 3/3 + decision 3/3 的完整治理链流转。本文记录过程中的可复用模式。
>
> 注："三轮独立检验"均在同一对话上下文中运行，独立性有限。真实的独立检验需不同会话或不同模型。本文记录的"独立"指"无交叉参考"，非"完全隔离"。

## 一、流程概要

```text
外部蓝皮书 (drafts/)
  -> iCL 认知：蓝皮书 x 现有体系发散诊断
    -> 九段反推检验 x 3 (C1/C3/C5)
      -> 多 Agent 独立对照 x 3 (每个冲突两份独立检验 + 一份对比裁决)
        -> 校准汇总 -> proposal 1/3 -> 审查修正 -> 2/3
          -> decision 1/3 -> 审查修正 -> decided-by -> 3/3
            -> proposal 3/3
```

全程耗时约 2 天，产出 6 份文档（1 proposal + 1 decision + 2 反推记录 + 1 蓝皮书底稿 + 1 本文）。

## 二、多轮独立检验的分歧模式

### 2.1 三种分歧类型

**类型 1：判定方向相反，校准终点趋同**

- 表现：两份报告给出相反的整体判定（通过 vs 不通过），但最终建议的校准方案几乎一致
- 示例：C3：一份说"有条件通过（验码）"，一份说"不通过"，但都认为应降级为"运行已有测试+结构化报告"
- 处理：取更严格判定，吸收两份报告的校准洞见

**类型 2：判定一致，论证互补**

- 表现：两份报告结论相同，但论证路径和发现的洞见不同
- 示例：C1：两份都说"有条件通过"，但一份用了类比检验（法官 vs 检察官），另一份用了相位分析（源头干预 vs 入口把关）
- 处理：合并论证，增加结论置信度

**类型 3：判定方向相反，校准终点也不同**

- 表现：两份报告的结论和方法都不同，需要第三份报告的裁决
- 示例：C5：一份说"不通过（替代：治理追溯标记）"，一份说"有条件通过（可信度评分）"。meta-review 裁决了方法论层面的分歧
- 处理：需要第三份独立审查 + 方法论层面的裁断

### 2.2 分歧根因

分歧的核心来源不是事实判断差异，而是**方法论预设**的不同：

| 预设维度 | 宽松端 | 严格端 |
|---------|--------|--------|
| "通过"的门槛 | 核心意图有价值，可通过校准保留 | 安全风险和职责越界是否可接受 |
| 校准的边界 | 九段式第九段可改变原主张的实体（如"认证"变为"评分"） | 校准应保留原主张的核心意图，不能做概念替换 |
| 道法侧重 | 损补优先：补足现有体系的缺失能力 | 知止优先：不引入无法控制风险的新能力 |

**本次案例的裁决顺序**（非通用规则，待更多案例检验）：

1. 先检查是否存在事实错误（事实优先于观点）
2. 再检查是否存在概念混淆（定义优先于论证）
3. 最后检查是否违反道法约束（知止优先于损补，假阳性代价高于假阴性）

### 2.3 有效模式：独立检验 + 对比裁决

蓝皮书案例验证了一个有效的治理模式：

1. **两轮独立执行九段检验**，不做交叉参考
2. **第三轮做对比裁决**：识别分歧、分析根因、裁定终局
3. **本次案例的裁决方式**：知止优先于损补。当两份报告结论冲突时，采纳更严格（更合知止）的判定，因为假阳性（错误引入）的代价高于假阴性（错误拒绝）。此方式在本案例中有效，**是否应成为通用规则需独立 proposal，不能从本 note 中产生**

这个模式使两份独立报告从"两个意见"变成了"一次交叉验证"：：当路径不同但终点一致时，置信度提高；当路径不同且终点不同时，需要 meta-review 裁断。

## 三、反推检验的方法论校准

### 3.1 九段式第九段的边界

蓝皮书案例暴露了九段式中"校准建议"（第九段）的边界问题：校准可以在多大程度上改变原主张？

**待检验假设**（非规则，本 note 无权修订 3/3 spec。如需修订鉴论，应走 Reopen 流程）：

> 第九段"校准建议"应在**保留原主张核心意图**的前提下调整边界。如果校准后的概念改变了原主张的治理对象（如从"认证 Agent"变为"评分 Agent 行为"），则这不是校准，而是概念替换。概念替换意味着原主张不通过，新概念是独立主张，需独立的检验。

### 3.2 证伪条件的元检验

C5 的检验中发现：证伪条件（"长期 Bug 率低于非治理代码"）在逻辑上可陈述，但在工程上**不可执行**（无法构造可比对照组、无法控制混淆变量）。

**待检验假设**（非规则，如需修订鉴论，应走 Reopen 流程）：

> 九段式第七段（可证伪条件设定）之后，应增加**证伪条件的可执行性检验**：条件是否可测量？是否可独立复现？是否可归因到被检主张？如果任一答案为否，证伪条件本身不满足"鉴"的实质要求，原主张应判定为不可证伪。

### 3.3 概念混淆的识别

蓝皮书中最隐蔽的错误不是事实错误，而是**范畴混淆**：

| 蓝皮书用语 | 司衡定义 | 混淆类型 |
|-----------|---------|---------|
| "鉴（反证）" = 沙盒测试 | 鉴 = 反推九段式（检验主张真伪） | 将认识论工具降格为工程工具 |
| "认证 Agent" | 治理对象是产出物（文档和代码） | 将治理对象从产出物转移到生产者 |

这提示：反推检验的第二段（概念分析）应特别警惕**同名异义**：：外部生成内容使用司衡术语但赋予不同含义。这不是恶意，而是自然倾向：：术语被借用但不理解其原始定义的精确边界。本次审阅中，外部审阅报告将陈述式语言读作经验记录，正是"同名异义"在审阅层面的实例化。

## 四、治理链的实战验证

### 4.1 Dev-Governance 治理链经受了完整测试

蓝皮书案例是 Dev-Governance 治理链（drafts → proposals → decisions → specs → src）的首次全链路实战：

- **drafts → proposals**：蓝皮书经 grill-me 追问（通过反推检验实现），满足转化门槛（gap 明确 + 方案对比 + 验收标准）
- **proposals 1/3 → 2/3**：结构化完成，进入审查
- **审查 → 修正 → 2/3**：发现 7 个问题（F1/F2/S1-S4/G1），全部修正后闭合
- **decision 1/3 → 2/3 → 3/3**：经两轮审查（格式 + 内容完整性），5 个缺口全部闭合，decided-by 签署

治理链证明了自己能处理外部生成的、与现有体系存在架构冲突的复杂提案。

### 4.2 发现的链路弱点

| 弱点 | 表现 | 建议 |
|------|------|------|
| 审查者的道四盲区 | 提案作者自己审查自己的提案，最初遗漏了 performative contradiction（提案要求 @limitations 但自己没有） | 审查者须与提案作者不同（已在 Dev-Governance 中规定，但单人项目中 AI 充任审查者时需额外警惕自指漏洞） |
| 修正闭环的追踪成本 | 7 个修正项的追踪状态需要显式记录，否则后续审查者无法判断是否已修正 | decision 中增加"修正项追踪"作为标准字段 |

## 五、@limitations

1. **单案例局限**：本文的模式总结基于一次实战。模式是否泛化需更多案例验证。
2. **"独立性"的有限性**：本次三轮检验均在同一对话上下文中运行，共享隐含预设。"独立"仅指"无交叉参考"，非"完全隔离"。真正独立的检验需不同会话或不同模型。本文不使用"多 Agent"表述，改用"多轮独立检验"。
3. **人类裁决的不可替代性**：最终 decided-by（moc）是人类。本文记录的"多轮独立检验"仍然在"AI 建议、人类决定"的框架内。Agent 之间的分歧裁决不能替代人类裁决。
4. **单人项目的结构性瓶颈**：本次治理链的每个环节（检验、审查、修正、裁决）都依赖同一个人类参与者（moc）做最终确认。在单人项目中，**人类始终是单点瓶颈**，无法完全消除。治理链的多角色分配机制仅在多贡献者项目中才能完全实现。
5. **审查者自指盲区**：本文作者同时是审查者。本文 §4.2 记录的"审查者盲区"（提案作者审查自己的提案遗漏自指漏洞）在本文的审阅中同样存在。本文可能遗漏了同样的自指盲区，需人类读者额外警惕。
6. **本文的证伪条件**：本文核心主张（P-meta）：蓝皮书案例中观察到的三种分歧类型和分歧根因（方法论预设差异），可泛化到其他多轮独立检验场景。如果在未来 3 次多轮独立审查中，分歧模式与本文的三种类型均不匹配（即出现本文未分类的新模式），则 P-meta 被证伪，本文的分类法应修正或废弃。
