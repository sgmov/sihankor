# id 格式连字符决策漂移

> 首次决策（260616 之前）确定 id 格式为 `YYMMDDHHMM[-NNN]-语义短名`（无连字符），并在全项目执行。实践后发现纯数字串对人类阅读有困难，决策漂移回有连字符格式。

## 漂移轨迹

| 阶段 | 格式 | 理由 |
|------|------|------|
| 初始决策 | `YYMMDD-HHMM` | 自然书写习惯 |
| 首次修正 | `YYMMDDHHMM` | 缩短 id，省一个字符 |
| 执行后实践 | `YYMMDDHHMM` | 发现 `2406101030` 比 `240610-1030` 难辨识，日期与时间的边界模糊 |
| 漂移决策 | `YYMMDD-HHMM` | 人类可读性优先于省字符 |

## 影响

- 已执行的无连字符 frontmatter id 需全部回退为有连字符格式（约 31 文件）
- 连字符决策是 id 格式的根本约束，应在 Document-Conventions 中固化
