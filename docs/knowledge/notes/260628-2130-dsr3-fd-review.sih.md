## 二、DSR-3 fd 关键发现

### 2.1 fd 与 bat 同作者、相似结构、不同治理姿态

fd 和 bat 都是 sharkdp 的 Rust 项目，都有 `doc/` 子目录（fd 的 doc/ 含 release-checklist.md、sponsors.md、fd.1；bat 的 doc/ 含 assets.sih.md、alternatives.sih.md）。但 bat 在文档侧主动拥抱了 `.sih.md` 格式，fd 则保留了纯 Markdown 文档体系。**这是项目作者治理意识的体现，不是司衡的适配问题**——同样是"无治理概念但有序"项目，bat 选择跨进司衡门内，fd 选择留在门外。

### 2.2 索引器对 fd 完全沉默

`discover.rs:40` 的硬筛选 `name.ends_with(".sih.md")` 使 fd 的 7 个 `.md` 文件（README/CONTRIBUTING/CHANGELOG/SECURITY + 3 个 .github/ISSUE_TEMPLATE + doc/release-checklist + doc/sponsors）全部被跳过。**fd 在司衡索引视角下是空集**。

### 2.3 修复普通 Markdown 对司衡运行时零影响

按用户指引"修复一个格式问题"——选了 CONTRIBUTING.md 第 42 行的无语言代码块（changelog entry 格式）添加 `changelog` 标签。修复后 `index_rebuild` 仍返回 `Discovered 0 / Parsed 0 / Indexed 0`。**这是 DSR-3 fd 的核心信号**：司衡运行时对 `.md` 文件的格式修复完全不可见。

### 2.4 数据库路径仍为全局 DB

DSR-3 bat 修复了 `find_db_path()` 的 `SIHANKOR_DB_PATH` 环境变量支持。DSR-3 fd 在跑 metrics 时观察到 `project_status` 返回的是司衡全局 DB（144 docs）的视图，**不**反映 fd 自身（0 docs）。这是因为 mcp_client.py 未设置 `SIHANKOR_DB_PATH`，所有跨项目度量实际上跑在同一 DB 上——与 DSR-3 bat 的"隔离 DB"做法不同。

### 7.5 度量管道的"信号-噪声"问题

两个项目都暴露了司衡度量管道在"跨项目基线采集"场景下的一个本质问题：**rule_audit/rule_density/tradeoff_coverage/trend_alignment 的数字反映的是 DB 状态，不是项目状态**。

- bat 用了隔离 DB（2 docs），所以这些数字反映 bat 自身（即使为空或极端值）
- fd 没用隔离 DB，所以这些数字反映司衡全局（144 docs），**与 fd 无关**

## 八、发现与洞察

### 8.1 "门外"项目的数据基线是空集

fd 的基线数字是 `Discovered 0 / Parsed 0 / Indexed 0 / Errors 0`——所有"零"都是同一信号：**项目不在治理范围内**。这不是司衡的失败，而是治理边界的清晰展示。

### 8.2 修复普通 Markdown 是"看不见的适配"

按司衡文档风格指南修复一个 CONTRIBUTING.md 代码块的语言标签，是治理姿态的表达，但**对治理运行时零影响**。这与"不修"在治理运行时的视角下没有区别——但对人类读者有意义。

DSR-3 fd 的 method_selection trail 明确记录了这个选择："故意不升级 .md 为 .sih.md"——保留"修复 vs 不修"的纯粹对比，让 snapshot_diff 显示 +0/+0/+0 的真实信号。

### 8.3 跨项目度量必须使用隔离 DB

DSR-3 bat 已经修复了 `SIHANKOR_DB_PATH` 路径隔离。DSR-3 fd 没设这个变量，导致 metrics 输出是全局 DB 的视图而非 fd 自身的视图。建议 mcp_client.py 增加 `--db-path` CLI 参数，并在 `--docs-dir` 改变时默认推断一个独立 DB 路径（如 `.sih/index-{project}.db`）。

### 8.4 同作者不同治理姿态的可对比性

fd 和 bat 是 DSR-3 提供的天然对照实验：
- 变量：作者的治理意识（是否把 .md 升级为 .sih.md）
- 不变量：作者（sharkdp）、语言（Rust）、规模（小型 CLI 工具）、目录结构（都有 doc/）
- 因变量：司衡可见文档数（bat = 2, fd = 0）

这是非常干净的对比——**差异来自项目特征而非司衡适配**。

## 十、下次 DSR 建议

1. mcp_client.py 增加 `--db-path` CLI 参数，默认根据 `--docs-dir` basename 推导独立 DB（如 `/tmp/dsr3-bat/` -> `.sih/index-bat.db`），避免全局 DB 污染
2. 当 docs_dir 下 0 个 .sih.md 时，project_status 给出明确提示"项目未进入治理范围（0 .sih.md）"而不是返回全局 DB 视图
3. 在 `infer_nature` 增加 `knowledge/trails/` 作为 trail 类型的标准路径，使 record_trail 写入的 trail 能被自动归类
4. 积累至少 5 个外部项目（其中至少 2 个用 DSR-3 fd 这种"门外"策略）的基线数据后，评估"门外"项目是否需要特殊的轻量级治理模式
5. 下次 DSR 周期尝试选择一个**主动升级 .md 到 .sih.md 的外部项目**（如其他已有 .sih.md 的 Rust CLI 工具），与 DSR-3 fd/bat 形成第三类对照
