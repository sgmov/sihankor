## 二、DSR-3 关键发现

### 2.1 索引器的 .sih.md 扩展名硬筛选

`discover.rs` 第 40 行硬编码 `name.ends_with(".sih.md")`，导致 bat 的 14 个 .md 文档全部被跳过。这是外部项目的首个治理屏障——没有 `.sih.md` 扩展名就没有入口。

### 2.2 数据库路径隔离缺失（已修复）

`main.rs` 的 `find_db_path()` 不读取 `SIHANKOR_DB_PATH` 环境变量，只能通过 CWD 向上查找 `.sih/index.db`。切换 docs_dir 时数据库不会自动切换，导致 `project_status` 返回旧项目的数据。

**DSR-3 中修复**：`find_db_path()` 增加 `SIHANKOR_DB_PATH` 环境变量支持。此修复在 DSR-3 中验证通过。

### 2.3 infer_nature 对非标准目录返回 unknown

2 个 `.sih.md` 文档放在 `/tmp/dsr3-bat/doc/` 下，`infer_nature` 返回 `"unknown"`（目录不在 `docs/specs/proposals/decisions/reference/knowledge` 标准树中）。这是已知 DSR-2 发现的重现。

### 2.4 trail 记录时的父目录问题

`record_trail` 推断 `knowledge/trails/` 路径时使用 `docs_dir.parent().join("knowledge/trails")`。对于 `/tmp/dsr3-bat`，行迹被写入 `/private/tmp/knowledge/trails/`。外部项目没有司衡的 `knowledge/` 目录，这本质上是一个未处理的外部场景。

## 七、发现与洞察

### 7.1 扩展名硬筛选是最外层治理屏障

对于外部项目，`.sih.md` 扩展名要求意味着要么改名现有文件，要么创建新文件。这不像 SiHankor-A（天然有 `.sih.md` 文件）那样可以无缝索引。

### 7.2 数据库路径隔离是跨项目部署的前提条件

DSR-2 中通过 `SIHANKOR_DOCS_DIR` 实现了文档目录隔离。DSR-3 暴露了数据库路径没有同步隔离。`SIHANKOR_DB_PATH` 修复后已验证可正常工作。

### 7.3 度量管道对陌生项目产生极端值

| 度量 | 外部项目值 | 说明 |
| ---- | ---------- | ---- |
| 规则密度 | 7.00 | 14 规则 / 2 文档 = 高密度（样本太少导致失真） |
| ADR 覆盖率 | 0.0% | 无 decision 文档，指标本身无效 |
| TrendAlignment | 1.00 | 审查=变更（2/2），同步但不一定有意义 |

### 7.4 MCP 客户端协议在跨项目场景下稳定

经过 3 个不同项目（司衡自身、SiHankor-A、bat），MCP initialize 握手、tools/call、通知处理均未出现协议级错误或超时。

## 八、建议

1. 增加 `SIHANKOR_DISCOVER_EXT` 或类似机制，使文档发现不再硬绑定到 `.sih.md`
2. 当 `knowledge/trails/` 目录不存在时，定义回退行为（如复制到 docs_dir 同级或报错）
3. 积累至少 5 个外部项目的基线数据后，评估是否需要独立的"轻量级治理"模式
