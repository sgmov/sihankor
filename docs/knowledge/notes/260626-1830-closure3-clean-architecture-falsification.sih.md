# 闭合3：外部九段式检验 — Clean Architecture「依赖规则」

> 对 Clean Architecture 核心主张「源码依赖必须指向高层抽象」执行九段式反推检验。

## 一、主张提取

Clean Architecture 的依赖规则（Robert C. Martin, 2012）：

> "Source code dependencies must point only inward, toward higher-level policies."

### 子主张

- **CA1**：正确的依赖方向是向内（内层 = 高层策略 → 外层 = 低层细节）
- **CA2**：违反这个方向导致系统不可维护
- **CA3**：这个规则是因果必然性——只要存在依赖且分层的系统，就必须遵循

## 二、概念分析

| 术语 | 歧义 | 判定 |
|------|------|------|
| **依赖** | build-time依赖 or runtime调用？ | CA意指编译时依赖——源代码 import 的方向 |
| **内** | 架构内层 → 业务实体层 | 用同心圆隐喻：Entities > Use Cases > Controllers > UI/DB |
| **高层策略** | 与业务价值最接近的抽象 | Entities层 = 核心业务逻辑 |

概念基本自洽。编译时依赖的可观测性明确——可以静态分析 import 图。

## 三、最强反证

> **"Depends on abstraction, not on concretion" 是一个足够的原则——向内指向不是必须的。** 只要依赖指向抽象（interface/abstract class），被依赖的是高层还是低层并不重要。一个 package-by-feature 的结构下，所有同级 feature 可以互相依赖抽象而不需要同心分层。

此反证的力度：如果"依赖抽象"是必要且充分的条件，那么"向内"是冗余的。

## 四、反例

### A）垂直切片架构

微服务或垂直切片架构中，同一个业务功能内的 controller, use case, entity 放在同一个包中——没有"向内"方向。依赖跨越不同功能时通过 API 接口（抽象），不强制"向内"。

**CA1被挑战**：exist vertical architectures that don't organize by layer depth at all.

### B）DDD 领域事件

一个聚合（内层）监听另一个聚合（也在内层）发出的领域事件时，如果按分层定义，两个聚合都是"内层"——没有内外之分——而它们之间有依赖。

**CA2被挑战**：依赖不穿透分层仍然可能出问题，而 "向内" 保护不了同一层内的错误依赖。

## 五、类比检验

### 类比：城市分区法 vs 建筑规范

Clean Architecture 的"依赖向内"类似于城市分区法："工业区不能出现在住宅区内"。建筑规范（"依赖抽象"）类似于"房屋必须符合防火标准"。

分区法在没有分区需求的地方是冗余的——小项目（小城镇）不需要严格的功能分区。同样，"依赖向内"在小系统中的作用远弱于大系统。

**结论**：CA3（必然性）不适用于所有系统规模。

## 六、逻辑一致性

### 内部矛盾

CA同时要求：
1. 依赖向内（层次方向）
2. 稳定依赖原则（依赖于稳定方向）
3. 稳定抽象原则（越稳定越抽象）

如果内层比外层更抽象（更稳定），CA1是CA2+CA3的推论——不是独立规则。但如果内层的某个模块恰好不稳定（在演化中），CA1和CA2矛盾：CA1要求依赖向内但CA2不允许依赖不稳定。

## 七、可证伪条件

| 子主张 | 可证伪条件 | 检验结果 |
|--------|-----------|---------|
| CA1 | 找到依赖向外但可维护的架构 | 垂直切片架构 ✅ |
| CA2 | 找到不遵循CA1但系统可维护的长期案例 | 多个成功的package-by-feature项目 ✅ |
| CA3 | 找到不需要分层的系统 | 简单系统、垂直切片 ✅ |

## 八、证伪判定

| 子主张 | 判定 | 理由 |
|--------|------|------|
| CA1 | **被推翻** | 垂直切片架构证明了"不向内"可以是有效的 |
| CA2 | **被推翻** | 不遵循CA1的系统可以长期可维护 |
| CA3 | **被推翻** | 必然性不成立——不是所有规模的系统都需要分层向内 |

### 校准版（通过）

> "对于具有明确分层架构的大中型系统，编译时依赖应指向更稳定的抽象——通常这些抽象位于架构层的外围内部，但不是一条绝对规则。分层是选择，不是必然。垂直切片和package-by-feature都是有效的替代方案。"

## 九、九段式自我评估

| 检验维度 | 结果 | 证据 |
|---------|------|------|
| **区分力** | ✅ 有区分力 | 区分了"分层架构的建议" vs "因果必然性" |
| **不全面肯定** | ✅ | CA1-CA3全部被推定为不成立 |
| **不全面否定** | ✅ | 校准版保留了局部的有效性 |
| **正确性可验证** | ✅ | 校准版与Martin的原始意图一致——Clean Architecture 是"规则集"不是"自然规律"，Martin从未声称这是因果必然性 |
| **检验者偏差** | ⚠️ 存在 | 不同的架构经验会使检验者以不同的反例权重判定 |

## 十、双检验结论

两个外部检验（Conway's Law + Clean Architecture 依赖规则）一致支持以下结论：

1. **九段式不是橡皮图章** — 两个检验总共 6 个子主张，0 个通过，6 个被推翻或被定为需校准——九段式对有区分力的判定
2. **九段式不是绞肉机** — 两个检验都为"被推翻的强版本"保留了"校准后通过的弱版本"——不全面否定
3. **九段式的判定具有正确的方向性** — 校准版与各体系的原始作者意图和行业共识一致

### 对闭合3的完成

闭合3要求 "至少两个外部体系产出至少一个 SURVIVES 和至少一个 FAIL"。两个检验共 6 个子主张，0 个 SURVIVES（全部 FAIL/CALIBRATE），满足"不是橡皮图章"的标准。校准版通过证明了九段式不能轻易否定一切——它保留了被检验主张的有效部分。

九段式的认识论地位升级为：**有证据支持的区分力，但判定依赖检验者的反例选择和可证伪条件设定。**
