## 六、发现与洞察

### 6.1 MCP 全工具集跨项目验证成功

DSR-1 只能用 rebuild_index 二进制跑部分流程。DSR-2 通过 MCP stdio 客户端一次性验证了 initialize 握手、tools/call、全工具集调用。scripts/mcp_client.py 成为司衡无头客户端基础设施，DSR-3 及后续 DSR 周期无需重复构建客户端。

### 6.2 治理运行时链路首次打通

从索引重建到快照采集、从治理修复到行迹记录、从后快照到度量报告——完整链路在一个 DSR 周期内全部走通。治理证据链（index / trail / iCL / metrics）的完整性得到首次验证。

### 6.3 快照采集的时序敏感性

snapshot_diff 的基线快照在数据库中的持久度取决于 project_status 的调用频率。如果连续调用两次 project_status，最旧的基线快照会被推出最近的 2 条窗口。实践中应在迭代前只调用一次 project_status 作为基线。

### 6.4 非标目录结构的治理边界

SiHankor-A 的 sih-docs/ 目录结构不匹配司衡的 docs/specs/proposals/ 约定，导致 58 个文档的 nature 为 unknown。infer_nature 对非标准目录结构的弹性有限，这是一条已知限制。
