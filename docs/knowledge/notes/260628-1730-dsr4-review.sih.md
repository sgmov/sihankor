# DSR-4 回顾：开源项目 ripgrep 外部治理基线采集

## 八、发现与洞察

### 8.1 文档体系大小与司衡可见性脱钩

ripgrep 的文档量是 bat 的 1.6 倍、fd 的 3 倍，GUIDE.md/FAQ.md/CHANGELOG.md 各自超过 1000 行——**但所有这些文档对司衡运行时完全不可见**。司衡可见性由 `.sih.md` 扩展名决定，不由 .md 行数决定。这是一个清晰的"入口与体量无关"的设计选择。

### 8.2 长格式文档的扫描表现一致

GUIDE.md 1025 行包含 92 个代码块，扫描时间在毫秒级（index_rebuild 总耗时未独立计时，但 22 个 .md 的 WalkDir 遍历 + 扩展名过滤是 O(n) 复杂度）。这验证了"大文档不被特殊处理"的假设——`discover_documents` 不做内容解析，只检查文件名后缀。

### 8.3 修复策略与文档量无关

ripgrep 有 46 个无语言代码块可选修复，但 DSR-4 故意只修 1 处——修复数量是策略选择，不是项目特征。bat 修 0 个代码块（修 1 个 10 列宽表），fd 修 1 个代码块，ripgrep 修 1 个代码块——三者的"修复数量"接近，都是"最小代表性修复"。

### 8.4 MCP 协议在三个不同规模项目上稳定

经过 3 个不同项目（bat 90MB、fd 15MB、ripgrep 30MB），MCP initialize 握手、tools/call、通知处理均未出现协议级错误或超时。stdio 通信在 DSR-3 bat 修复的 `SIHANKOR_DB_PATH` 隔离路径下，DSR-4 第二次独立验证。

### 8.5 "门外"项目策略可推广

DSR-3 fd 验证了"门外"策略（不升级 .sih.md）的纯粹性。DSR-4 ripgrep 在更大文档体系下重现了相同结果：

| 验证项 | fd | ripgrep | 一致性 |
| ------ | -- | ------- | ------ |
| Discovered/Indexed 修复后 | 0/0 | 0/0 | 一致 |
| Snapshot diff Documents | +0 | +0 | 一致 |
| 隔离 DB 视角 | （无） | 0 docs | 一致 |
| 普通 Markdown 修复不入索引 | 是 | 是 | 一致 |

这意味着"门外"策略可作为司衡外部验证的默认模式——与"升级"模式并列，二者覆盖不同的治理姿态表达。
