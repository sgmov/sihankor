# C1/C3 反推检验记录：蓝皮书冲突主张的合道性判定

> 两份独立检验记录及其对比分析。检验对象：蓝皮书 C1（主动监察主权）和 C3（沙盒反证）。
> 检验方法：反推九段式（依据《司衡鉴论》$二）。

## 一、C1（主动监察主权）检验摘要

**被检主张**：司衡应具备后台监察 Agent，主动监听文件系统事件、在 PR 下评论、阻断不合规合并。

**整体判定**：**有条件通过。**

**关键校准**：

| 校准点 | 内容 |
|--------|------|
| 阻断实现 | 司衡返回 exit code，CI 平台执行阻断；司衡不直接操作 PR 状态 |
| 阻断规则 | 仅可形式化规则（AST/禁止列表）可阻断；语义判断仅 warn |
| 触发方式 | 事件驱动（CI webhook），非定时轮询 |
| PR 评论 | 结构化报告（规则编号+位置+修复建议），非自由文本 |
| F/G/J 过滤 | F 级：阻断+评论；G 级：评论不阻断；J 级：静默记录 |
| 证伪阈值 | 虚报率 >30% 或忽略率 >50%，6 个月后收紧至 20% |
| 注意力屏障 | 主动监察消除了被动检查的天然门槛，需明确替代节制机制（来源：另一份报告的贡献） |

**校准后命名**：**CI 守护**（替代"后台监察 Agent"）。

**道法映射**：事件驱动（有度），可形式化规则才阻断（知止），语义判断仅 warn（顺因），CI 平台执行阻断（知止），结构化报告（损补）。

## 二、C3（沙盒反证）检验摘要

**被检主张**：司衡应在沙盒中运行动态反证，注入边界/异常输入，尝试使代码崩溃或偏离规约，并根据结果决定是否回退。

**整体判定**：**有条件通过。**

> 注：另一份独立报告判定为"不通过"。分歧分析见 $三。

**关键校准**：

| 校准点 | 内容 |
|--------|------|
| 正名 | **验码**（非"鉴·反证"）。鉴检验主张（反推九段式），验码检验代码鲁棒性（动态测试）。两者不可混淆 |
| 执行内容 | 运行项目已有测试+集成现有 fuzz 工具（cargo-fuzz/proptest），不自动生成边界输入 |
| 判定范围 | 仅判定 crash/panic/超时，不判定"规约偏离"（语义判断不可自动完成） |
| 回退机制 | CI 失败→阻断合并（经 C1 校准后），不自动触发代码重生成 |
| 与 code-lint 关系 | 互补：code-lint（静态规则违反） + 验码（动态鲁棒性） = 双层代码质量验证 |

**道法映射**：正名验码≠鉴（顺因），集成现有工具（知止），仅判定 crash/panic（知止），CI 阻断非自动重生成（知止+顺因）。

### 2.1 核心发现

蓝皮书将 fuzz testing + 单元测试统称为"鉴（反证）"，这是**正名错误**。鉴有确切含义：反推九段式、诗理区分、可证伪条件。代码鲁棒性测试不是鉴，正如望远镜不是天文学。

C3 校准后与现有体系形成三层验证：

| 层 | 工具 | 做什么 | 司衡中的名称 |
|---|------|--------|-------------|
| 静态规约验证 | code-lint (clippy) | AST 级别的规则违反检测 | 术·验 |
| 逻辑治理验证 | iCT | 五法校验、道法一致性 | **鉴** |
| 动态鲁棒性验证 | 验码（C3 校准后） | 沙盒中运行已有测试+fuzz | **验码** |

## 三、C3 两份独立检验的对比分析

### 3.1 分歧总览

| | 本报告 | 另一份报告 |
|---|---|---|
| 整体判定 | 有条件通过 | 不通过 |
| P1（沙盒执行） | 有条件通过 | 不通过 |
| P2（边界输入） | 有条件通过（集成现有工具） | 有条件通过（降级为运行已有测试） |
| P3（自动回退） | 有条件通过（严格校准） | 不通过 |
| P4（鉴的必要部分） | 通过（正名为验码） | 不通过 |

### 3.2 分歧根因分析

**分歧 #1：对"通过"门槛的定义不同。** 本报告先问"核心意图是否有价值，能否通过校准保留价值"；另一份报告先问"安全风险和职责越界是否可接受"。两种姿态导致同一事实集合得出相反结论。

在"知止"意义上，另一份报告更正确；在"损补"（补足 code-lint 无法覆盖的鲁棒性验证）意义上，本报告更正确。两者都是司衡法的有效维度，没有绝对对错。

**分歧 #2：P1 沙盒安全论证。** 另一份报告以"沙盒逃逸"为否决理由之一。但 CI 已经在运行 `cargo test` 和宏展开（宏展开就是任意代码执行），蓝皮书 C3 没有引入 CI 原本没有的攻击面。本报告在此点上更准确。

**分歧 #3：P4 方法论差异。** 另一份报告认为原主张"反证是鉴的必要部分"是错的，校准后的"验码有价值"是另一个主张。本报告认为九段式第九段"校准建议"的存在意义就是允许主张在检验中变形存活。本报告的方法论更合九段式的设计意图。

### 3.3 强一致点

两份报告在以下判断上完全一致：

- "反证失败 != 规约偏离"：崩溃可能是环境/时序/输入问题，不是意图偏离
- "鉴 != fuzz testing"：混淆范畴
- "自动回退不合知止"：代码生成是外部 Agent 的因果链

两份报告用不同论证路径抵达同一校准终点，提高了终局的置信度。

### 3.4 结论

C3 判定为**有条件通过**（本报告立场）。另一份报告"不通过"的严格精神是更合知止的，但"有条件通过"在损补意义上更完整。两份报告的校准终点趋同：运行已有测试+结构化报告保留，自动生成输入+语义判定+自动回退移除。
