### Commit Hygiene 经验（两条）

#### 经验 #1：commit 前检查分支

```bash
git branch --show-current
```

预期输出：当前工作分支的纯名字（如 `index-stale-prune`）。如果输出是 `main` 或别的分支名，立刻停下来重新审视本次 commit 该往哪个分支落。

#### 经验 #2：commit 后检查 message

```bash
git log -1 --format='%s'
```

预期输出：自己刚才填写的 commit message 第一行（如 `docs: rebuild_index stale-prune pass proposal`）。如果 message 是别人的（如本次事故中出现的 `ratify: techne-orchestration...`），说明 commit 跑到了别人的工作上下文，需要立刻检查 reflog 找回自己的 commit。

### 为什么是这两条而不是更多

不引入第三条、第四条 hygiene 检查，原因：

- 多 session 并行交错的核心问题是**身份确认**，不是流程细化。加更多"先 X 再 Y 再 Z"的步骤只会增加 agent 的认知负担，不解决根因。
- `git branch --show-current` + `git log -1 --format='%s'` 已经构成最小可执行的身份双因子：分支 + message。两者一致则 commit 是你的，否则立刻能发现。
- 这两条检查成本极低（每个 < 100ms），可以无脑嵌入 commit 前后的 shell 习惯。

### 不改 AGENTS.md 的理由

并行 session 的 git 操作发生在执行环境层面（host 上的 git daemon、文件系统锁、agent 调度），AGENTS.md 是文档规范，无法约束执行环境的行为。如果把"并行 session 必须串行"写进 AGENTS.md，反而会让单 session 流程变得臃肿。

有效防护只能靠 agent 自律：每次操作前确认"这条命令会在哪条分支上执行"，操作后确认"这次 commit 的产物是不是我预期的"。
