# DSR-1 行迹记录

> DSR-1 过程中的方向性转折点，作为行迹机制的首批数据原型。当前 6 条（T1-T6）。

## T1：修复顺序抉择

**转折**：原计划在 DSR 之前做工程集成修复。

**理由**：交叉审阅发现工程映射中三个可修复的集成缺失（V-F-03、decided-by、环检测），堵漏优先于冒险。DSR 的数据基础需要诚实化后才能产生可信验证数据。

**后果**：DSR 启动晚了一个迭代周期，但基线数据的质量更高——不会因 V-F-03 失效而漏过终态文档。


## T2：度量管道未激活

**转折**：原计划用 MCP server 采集完整的 `ProjectSnapshot` + `VarianceMetric`，实际只用了 `rebuild_index` 二进制。

**理由**：MCP server 需要 stdio 交互，在当前会话中无法同时运行 MCP server 和操作 agent。`rebuild_index` 提供了索引管道的最小可用数据（discovered/parsed/indexed/errors），满足了 DSR-1 的核心需求。

**后果**：DSR-1 只有索引级对比，缺少违规级方差度量。DSR-2 需要通过 MCP 客户端或脚本启动 server 来采集完整度量。


## T3：SiHankor-A Stage 格式发现

**转折**：预期 SiHankor-A 的文档格式应与司衡完全兼容，实际发现 stage 字段使用 `propose`/`ratify` 而非 `1/3`/`2/3`/`3/3`。

**理由**：SiHankor-A 有自己的治理生命周期概念，语义等价但编码不同。这不是"错误"，是跨治理系统的格式间隙。

**后果**：39 个索引错误中 25 个属于此类型。这是 DSR-1 最有价值的发现——验证了道二（编码必有损）和道四（规约与实现必有间隙）。


## T4：目录映射的临时方案

**转折**：SiHankor-A 使用 `sih-docs/` 而非 `docs/` 作为文档根目录，`rebuild_index` 硬编码 `docs/`。

**理由**：`ln -s` 和 `mv` 因文件系统限制均不可用。最终采用 `cp -r` 到工作区临时目录的方案。

**后果**：可行但不优雅。暴露了索引配置的缺失——`docs_dir` 应该是可配置的，不是硬编码。这可能是未来工程改进的方向。


## T5：Archive 的 Nature 推断缺口

**转折**：`archive/` 目录下的文档无法被 `infer_nature` 识别，nature 推断返回 None。

**理由**：SiHankor-A 有三个版本的 archive（v1/v2/v3），内容与活跃文档的 stage 格式相同但 nature 无法推断。当前 `infer_nature` 没有为 archive 目录定义映射。

**后果**：虽然 nature 推断失败不阻断索引，但依赖 nature 的规则（如 V-F-04 upstream 必填性检查）可能产生误判。archive nature 是未来 `infer_nature` 扩展的候选。


## T6：行迹字段转向原型验证

**转折**：行迹提案原设计使用 `from_state → to_state` 的字段拆分来记录转折，但录入 T1-T5 时发现该拆分不实用。

**理由**：现实决策中"转折前状态"和"转折后状态"难以干净分离。T2（度量管道未激活）是方法选择，没有明确的"前"和"后"；T3（Stage 格式发现）是跨系统发现，不属于转折。设计驱动的精确保留假设了现实转折可以被干净地二分，原型录入证明这个假设不成立。

**后果**：数据结构从 `from_state + to_state` 合并为 `turning_point`，新增 `consequences` 字段记录决策的连锁反应。新增 `discovery` 类型覆盖非转折类发现。这一修正说明：先做手工原型再做工具设计，比先设计工具再适配数据更高效。
