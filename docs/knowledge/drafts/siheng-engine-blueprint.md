## ——基于"道法术鉴约形迹"的MCP基座与Agent编排系统

> 此为草稿，不遵守 .sih.md 格式。待与现有司衡体系推演碰撞后，产出的正式文档将按治理规约格式化。

### 0. 文档声明与"鉴"（可证伪性条件）

本声明遵循司衡哲学之"鉴"：

> 本蓝皮书所描述的系统，若在实施后无法达成以下目标，则本设计视为被证伪，应被修正或废弃：
> 1. 治理介入后，AI生成代码的长期维护成本（以Bug率/理解时间衡量）显著低于非治理代码。
> 2. 在标准测试集上，司衡引擎能阻断>=95%的已知意图偏离模式。
> 3. 司衡引擎自身的缺陷密度，不高于其所治理的项目平均值。

本蓝皮书不是神圣教条，而是一组等待被攻击、被修正、被证伪的**工作假设**。

### 第一部分：哲学基石（本源）

#### 1.1 核心命题

| 名称 | 命题 | 一句话 |
| :--- | :--- | :--- |
| **道一** | 发散自-然，收敛必-为 | 发散会自动发生，收敛必须有意识地执行。 |
| **道二** | 意图先于代码 | 代码是意图的有损编码，因果方向不可逆。 |
| **道三** | 代码自晦，意图必复 | 代码不会自行揭示其意图，维护前必须恢复。 |
| **道四** | 规约与实现必有间隙 | 治理引擎自身也受道的约束，可能出错。 |
| **五法** | 顺因-有度-知止-损补-顺势 | 在收敛时，尊重因果、分级力度、克制干预、动态平衡、适配场景。 |
| **六层脉络** | 元->道->法->术->几->约->形迹 | 从元公理出发，逐层收敛为可执行的规约与代码，最终沉淀为历史经验。 |

#### 1.2 "鉴"：：认识论保障

> **所有道层主张必须指明可证伪条件。证伪必须基于可独立复现的真实工程案例。**

换言之，司衡拒绝任何不可被推翻的宣称。所有设计决策都必须附带"如果X发生，则本设计失败"的退出条款。


### 第二部分：核心定位：：司衡不是什么，是什么

| 司衡不是... | 司衡是... |
| :--- | :--- |
| 一个代码生成器 | 一个代码生成的**治理者** |
| 一个独立的AI Agent | **Agent的Agent**（元治理层） |
| 一个被动的MCP工具库 | 一个**主动监察+被动服务**的双重实体 |
| 一个本地大模型 | 一个**轻量级确定性状态机**（< 30MB核心） |
| 一个哲学宣言 | 一个**可证伪的工程系统** |

**一句话定义**：司衡是运行在AI Agent与代码仓库之间的**治理滤网**：：它确保每一个代码变更都经过了"意图对齐->策略分级->自我反证->历史反思"的完整流程。


### 第三部分：系统架构：："监察者+服务端点"双重身份

司衡引擎运行时的物理形态为**一个常驻进程**，同时扮演两个角色：

#### 角色A：后台监察Agent（主动）
- **感知**：监听文件系统事件、Git Webhook、PR变更。
- **决策**：基于确定性规则引擎（AST匹配、哈希对比、规则推理）扫描意图偏离与违规。
- **行动**：在PR下发布评论、更新形迹日志、触发告警、阻断不合规合并。
- **约束**：**绝不拥有代码写权限**。所有告警和阻断均可被人类手动覆盖。

#### 角色B：MCP服务端点（被动）
- **工具清单**：暴露 `check_intent`、`measure_drift`、`generate_recovery`、`retrieve_trail` 等只读工具。
- **响应**：接收外部Agent的调用请求，返回确定性判定结果或结构化建议。
- **降级模式**：当后台监察发现外部Agent跳过治理流程时，MCP返回结果自动附带 `governance_freshness: "stale"` 标记。

#### 二者协同闭环
```
外部Agent调用MCP工具 -> 拿到意图摘要 -> 生成代码（可能跳过check）
-> 后台Agent感知到未标记代码变更 -> 自动运行漂移检测
-> 漂移超标 -> 在PR下发布治理告警 -> 外部Agent看到告警 -> 重新调用MCP工具修正
```

### 第四部分：核心流程：："道法术鉴约形迹"状态机

司衡引擎内部运行一个**确定性状态机**，所有Agent的代码生成请求都必须经过此状态机才能被认可为"已治理"。

| 阶段 | 动作 | 产出物 | 是否依赖LLM |
| :--- | :--- | :--- | :--- |
| **1. 立道** | 将用户/Agent输入意图转化为结构化 `{Goal, Constraint, Uncertainty}` | `Intent_V1` | 是（首次转化） |
| **2. 定法** | 根据风险等级（资金/并发/权限）匹配F/G/J三级力度 | `Policy_V1` | 否（规则匹配） |
| **3. 生术** | 生成代码草案（可由外部Agent生成，司衡仅验证） | `Code_Draft` | 否（外部提供） |
| **4. 鉴（论证）** | 静态分析：验证代码是否满足所有约束（意图锚点、复杂度阈值） | `Evidence_List` | 否（AST分析） |
| **5. 鉴（反证）** | 动态测试：在沙盒中注入边界/异常输入，尝试使代码崩溃或偏离规约 | `Falsify_Report` | 否（沙盒运行） |
| **6. 鉴（决策）** | 若反证成功（代码被攻破），回退到阶段3；若反证失败（代码通过攻击），进入阶段6 | `Verdict` | 否（循环终止判定） |
| **7. 约/形迹** | 提取代码中的关键参数回写至规约；将本次反证记录追加至形迹日志 | `Intent_Final` + `Trail_Entry` | 否（结构化写入） |

**超时与熔断**：单次反证循环若超过60秒，或沙盒内存占用>500MB，状态机自动终止本轮推理并上报人类。


### 第五部分：工程实施路线图（三阶段）

#### Phase 1：人治垫片（0-1个月）：：立即生效
- **实施方式**：利用现有Agent的System Prompt与 `.cursorrules` 文件，手动植入"道法术鉴"流程；使用Bash/Python垫片模拟 `check_intent.sh` 和 `append_trail.sh`。
- **目的**：积累人工治理经验与形迹数据，为后续自动化提供需求规格与测试基准。
- **验证标准**：收集100个由垫片流程生成的代码样本，记录其后续Bug率与维护成本。

#### Phase 2：Rust核心状态机（1-3个月）：：核心引擎
- **实施方式**：
  - 用Rust实现"鉴"的确定性规则引擎（基于 `tree-sitter` 静态分析 + 沙盒动态测试）。
  - 编译为单一可执行文件（< 20MB），提供 `siheng-agent check`、`siheng-agent prove`、`siheng-agent ci` 子命令。
  - 同时暴露MCP Server端点（Stdio + HTTP/SSE）。
- **依赖**：外部LLM（仅用于阶段1"立道"的意图结构化转化；可对接Ollama或云端API）。
- **验证标准**：在内部测试集（含100个已知意图偏离样本）上达到>=95%的拦截率。

#### Phase 3：MCP基座与Agent编排网络（3-6个月）：：生态整合
- **实施方式**：
  - 发布VSCode插件，内置司衡MCP Client，自动发现并启动本地 `siheng-agent` 进程。
  - 实现多Agent协同协议：司衡作为仲裁者，接收不同Agent的方案并基于形迹历史进行裁决。
  - 开放"治理适配器"，允许第三方Agent接入司衡网络。
- **验证标准**：在至少3个真实开源项目中部署司衡，收集其治理数据并与Phase 1基线对比。


### 第六部分：外部依赖与体量控制

| 组件 | 体量 | 依赖 |
| :--- | :--- | :--- |
| 司衡核心二进制（Rust编译） | < 20 MB | 无（静态链接） |
| MCP适配层 | < 2 MB | 无 |
| VSCode插件 | < 5 MB | 无 |
| 外部LLM（可选） | 0（仅API调用） | 由用户自备（Ollama/OpenAI/Claude） |

**原则**：司衡引擎**绝不内置任何模型权重**。LLM仅用于"意图结构化"这一语义入口，且输出受JSON Schema严格约束，可被Rust类型系统强制校验。


### 第七部分：部署形态与用户集成

| 使用场景 | 部署方式 | 用户操作 |
| :--- | :--- | :--- |
| **本地开发** | VSCode插件自动启动后台进程 | 安装插件，无感知使用 |
| **CI/CD流水线** | `siheng-agent ci --path ./src` | 在GitHub Action中调用，阻断不合规PR |
| **MCP调用** | `npx @siheng/mcp-server` 或 `siheng-agent serve` | 供Cursor/Trae/Codex等Agent动态发现并调用 |
| **离线模式** | 仅运行确定性检查器，不连接任何LLM | 适用于内网或高安全场景 |


### 第八部分：哲学思辨认证：：如何声称Agent"具有司衡思辨"

当一套Agent或MCP流程完整通过了上述状态机（尤其是"鉴"的反证环节）后，可被授予 **"司衡思辨认证"**。

**认证声明（示例）**：

> *"本Agent/系统已通过司衡治理框架认证，在其行为层面表现出与司衡哲学预测一致的'意图对齐->策略分级->自我反证->历史回响'四维特征。此认证不涉及意识或主观体验，仅限于对可观测行为的断言。该断言接受如下证伪条件：若在持续维护中，其产出代码的长期Bug率未显著低于非治理代码，则本认证自动失效。"*

**认证不是奖励，而是一个附带证伪条件的工程契约。**


### 第九部分：风险与应对（道四的工程化）

| 风险 | 应对 |
| :--- | :--- |
| Agent绕过司衡直接改代码 | 后台监察进程定期扫描文件系统与Git历史，发现未标记变更则告警/阻断（见第三部分） |
| Agent伪造治理令牌 | 非对称签名+动态随机Nonce挑战+时间窗绑定（见前序讨论） |
| 司衡自身出现Bug（道四） | 熔断机制（告警频次超阈值自动降级）；独立审计委员会审阅核心规则变更；形迹日志公开可查 |
| 外部LLM输出不可控 | JSON Schema强约束；解析失败自动重试（2次）；降级为纯确定性提示 |


### 第十部分：总结：：司衡的终极愿景

> **司衡不是为了让AI"更聪明"，而是为了让AI"更可信"。**

当这套系统完全落地时，AI编程将从依赖"大模型的概率输出"转变为遵循"治理逻辑的确定性收敛"。开发者信任AI代码，不再是因为模型参数够大，而是因为每一行代码都经过了 **"意图-策略-反证-记录"** 的完整检验，且这份检验记录**公开、可查、可证伪**。

这就是司衡引擎的最终使命：**成为AI原生时代，代码世界的第一部"治理宪法"。**
