## 强制约束

1. **禁止使用判断性评价语言，但允许功能性描述。** 禁用词：缺陷、失败、成功、优秀、糟糕、正确、错误。允许的功能性描述： "在维度 D 上的覆盖范围是 X 到 Y"、"该位置决定了它能回答 Z 类问题，但无法回答 W 类问题"、"在该位置上，体系的知识产出边界由 X 约束"。核心区别：你描述能力边界，不做价值判断。

2. **必须从以下维度逐一定位：**
   - 认识论立场： empiricism / rationalism / pragmatism / constructivism / 其他？
   - 方法论：演绎 / 归纳 / 辩证法 / 设计科学 / 其他？
   - 治理哲学： top-down / bottom-up / adaptive / emergent / 其他？
   - 验证机制：内部自洽 / 外部经验 / 社会共识 / 实用效果 / 其他？
   - 知识主张类型：描述性 / 规范性 / 分析性 / 混合？

3. **必须与至少以下 5 个已有框架做对比定位：**
   - Design Science Research (Hevner et al. 2004)
   - Lean/Agile 软件开发方法论
   - Conway's Law 相关的组织治理理论
   - Cybernetics (Wiener) / Software Cybernetics
   - 任何你认为最相关的哲学传统（你自己选择，说明选择理由）

   每个对比必须回答：
   - 在哪些维度上与司衡重叠
   - 在哪些维度上司衡有独特位置
   - 重叠度估计（高/中/低）

4. **必须构建一个简化的知识图谱描述：**
   用文字描述一个二维或三维的空间，标出司衡和上述 5 个框架的位置。不需要图形，用坐标或方位描述即可。例如："在'内部验证 -> 外部验证'轴上，司衡位于 [偏内部 70% 处]，DSR 位于 [50% 处]..."

5. **必须回答一个核心定位问题：** 如果把软件工程治理的知识空间比作一个地图，司衡占据的是：
   - 一块已被其他框架覆盖的领地？
   - 一块未被覆盖的空白区域？
   - 其他框架之间的桥梁地带？
   - 还是其他位置？
